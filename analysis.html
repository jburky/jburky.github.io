Analysis

So, how do we know if we have Big Data problems? Big Data challenges can be categorized into three areas:

  • Volume
  • Velocity
  • Variety
Volume refers to having too much data and having a hard time managing it. Velocity issues come from a large influx of data. Variety problems comes from trying to integrate a large number of data sources. We will take a look at these areas and see what technologies exist today to help alleviate these challenges and how these technologies could help us in our proposed use-case, Community Medical Response (CMR). To support this future scenario, we will need to gather, manage, and process enormous amounts of data. I'll explain along the way.

Data Collection

For CMR we would also need to ingest, validate, transform, correct, and consolidate all of that data in a scalable way with minimal data scientists. One data preparation tool that exists and could help scale the work of a single data scientist is Data Wrangler. This tool assists the user by suggestive based transformation. It shows the user a quickly digestible sample of the data where the user can indicate artifacts for Data Wrangler which in turn suggests transformations. The user can preview the suggested transforms on the sample data and apply the best transforms that fits their needs. This reduces the barrier of entry and allows a much faster turnaround when transforming data by a single data scientist. In addition, we'll need a way to process a large number of data sources at scale and help alleviate the load on data scientists. The tool for this job would be something similar to Data Tamer (Tamr). Data Tamer (Tamr) does this by inverting the normal ETL architecture. For the different data sources Data Tamer ingests or loads each data source, normalizes multiple schemas into one global schema, de-duplicates the data among the different sources, and provides a visually guided data correction and transform tool that's similar to Data Wrangler. Instead of a data scientist solving these problems and telling the machine how to integrate each data source, the computer uses machine learning to solve these problems. Using machine learning algorithms to curate the data, the system would only ask humans for help via targeted crowd sourcing if there were discrepancies that it can't reconcile based on a certainty threshold. Tools like these would help in aggregating all of the data sources to support CMR. To support medical responses that are predictive and reactive we would need to collect data from sources like: medical records, eating habits, driving habits, locations of medically trained professionals for dispatch, vitality sensors, current events and individual locations relative to these events.

To support a mission critical service like the CMR system we would need high availability, on demand load balancing and elasticity, all at an affordable price. I would equate this to the importance of an electrical generator in a hospital during a power outage. Load balancing and elasticity would be important because there could be cases of increased usage when processing alerts and coordinating deployments of professionals. For example, a Friday night would most likely have higher rate of issues and generate more requests to the system versus a Sunday morning. We would also need this infrastructure at the lowest price possible so the service could operate for all communities. Let's look at how the Cloud could support these needs.

How could the CMR system stay cost effective while managing larger data sets, high computational throughput on demand, and remain reliable? A current approach to this problem is to use Cloud based services. Cloud storage and computing benefit customers with fast deployments, outsourced management, lower cost, with elasticity. Cloud storage allows customers to pay as they go removing the overhead for implementing their own enterprise level storage. Cloud computing also allows customers to scale up resources on demand as they are needed. These benefits are possible from Cloud service providers through economies of scale, variable utilization, and elasticity. Economies of scale reduces cost by spreading out the overhead cost of hosting. For example, if four separate companies hosted their own servers they each would incur the overhead cost of things like administration, electricity, high availability, security, and space. But if all four companies were hosted using the same provider they could split the overhead cost and reduce each individual overhead cost by 70%. This would allow the provider to also profit on a 5% margin from each client's savings. Additional cost savings come from variable utilization, which helps spread the resource requirements for each hosted client over time. If we put client A and B on the same server when client A only has peak usage during the day and client B only has peak usage during the night then that server if fully utilized increasing the ROI for that server. Elasticity allows clients to run bursty computations when working set can be done in parallel. There is no additional cost to run 100 servers for 1 hour versus 1 server for 100 hours. Cloud hosting provides other cost saving options such as: Software as a Service (SaaS), Platform as a Service (PaaS) and Infrastructure as a Service (IaaS). Software as a Service provides turnkey applications ready for client use and is one the service models that benefits the most from economies of scale. Platform as a Service provides ready to use developer stacks. Infrastructure as a Service provides raw computing resources when fine grained control is needed when running your computations or services. The benefits from Cloud hosting could help meet the needs of the CMR system.

Data Storage

Traditional databases would not be able to support the CMR system. It would likely need to be upwards of 100 times faster than traditional databases. We would need a data warehouse for analytics to keep track and predict medical needs in communities, and a transactional database for atomically dispatching medical personnel or equipment so we don't dispatch multiple of each on accident. Luckily, modern databases are well on their way to achieving this level of performance for each of these database types. For extremely fast OLTP databases we could use array based database systems. Array based database systems increase performance by storing all records in memory and removes the need for the DBMS to directly support slow operations such as latching, recovery, locking, and buffer pooling. This allows the DBMS to spend more time on the useful work load. For high performance analytics we would want to use data warehouses based on column store structures and organized in a star or snowflake schema. Column executors on array based database systems use vector processing which is wildly faster than row executors. It also supports compression in an easier and more productive way. Current implementations are a factor of 50 to 100 times faster than traditional databases. Continued performance increases like these will support the needs of the CMR system in the future.

A single Cloud services alone will not be enough for our CMR system needs. To capitalize on the elasticity of the Cloud that allows for bursty computations we will need a more distributed approach to our data processing. Technologies similar to MapReduce would allow us to spread our processing among multiple nodes in the Cloud. This processing needs to be broken down into map and reduce functions. Map functions process data independently and in parallel without dependencies on data processed at other nodes. Reduce functions run on one node and aggregate all the data processed by prior map functions. Techniques like this will help our CMR system scale.

One of the most important aspects of the CMR system is the required security for all of this data collected. If we were to use technologies like the Cloud and distributed computing we would want to encrypt the data at rest but still need an efficient way to process, analyze, and query the data. CryptDB is an example of technology that uses onion layer encryption which minimizes the impact to performance and maintains a relatively high level of encryption. Onion based encryption supports multiple encryption schemes like semantic security, homomorphic encryption, searchable encryption, deterministic encryption, and order-preserving encryption. This allows for multiple operations on encrypted data like joins, equality, sorts, and ordinal filters. It does this and remains secure by only peeling back encryption layers as determined by the query requested. Additionally, you can limit leakage by setting thresholds where only certain columns support certain query operators. This type of security would be need to ensure that the CMR system remain secure while holding sensitive information like medical records, driving statistics, and other personal information.

Data Analytics

The CMR system would need to support near real-time analytics to provide the fastest medical response possible for a person in need. Fast algorithms like streaming would be a good option for monitoring sensor data from each community because no data element is missed. While a fast algorithm like sampling may be better for QA analytics that could tell us if we are sending the right medical professional on the quickest route. This could be used to help adjust and tune other algorithms in the CMR system.

Another possible source of data for the CMR system would be social data like tweets, text messages, and phone calls. Using machine learning like natural language processing could help trigger medical response without the need to dial 911 manually. Imagine you're talking on the phone with a friend when they tell you they're having a heart attack. The CMR system could trigger on this information along with other sources of information and send help without every needing to make an additional call.
